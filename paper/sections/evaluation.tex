\section{Evaluation}
\label{sec:evaluation}

\subsection{Experimental Setup}

All experiments were run on a single machine (Windows 11, Python 3.14)
with both engines using in-memory SQLite storage.  Each data point
is the average of 10 warm-cache runs after 2 warm-up iterations.
The synthetic dataset generator creates facts with controlled arity,
context depth, and entity distribution (seed 42).

\subsection{Insert Performance}

Both engines achieve comparable insert performance, with SheafDB
slightly faster due to its simpler storage model (no triple decomposition).
Figure~\ref{fig:throughput} shows insert latency at three scales.

\begin{figure}[htbp]
\centering
\includegraphics[width=0.6\textwidth]{figures/throughput.pdf}
\caption{Insert throughput: SheafDB matches KG across all scales
(0.88--0.96$\times$).  Ratio annotated above bars.}
\label{fig:throughput}
\end{figure}

\subsection{Query Performance}

Figure~\ref{fig:latency} compares LOOKUP and GLOBAL query latency.
SheafDB's LOOKUP is orders of magnitude faster because it uses a
hash-indexed stalk for O(1) fact retrieval.  GLOBAL queries (full scans)
are faster on KG because the Sheaf engine iterates over all open sets.

\begin{figure}[htbp]
\centering
\includegraphics[width=\textwidth]{figures/latency.pdf}
\caption{Query latency (log scale).  SheafDB LOOKUP is 3--30$\times$
faster; GLOBAL is 1--1400$\times$ slower due to open set iteration.}
\label{fig:latency}
\end{figure}

\subsection{Speedup Analysis}

Figure~\ref{fig:speedup} shows the SheafDB/KG latency ratio.
Values below 1 indicate SheafDB is faster; above 1 indicates KG is faster.

\begin{figure}[htbp]
\centering
\includegraphics[width=0.6\textwidth]{figures/speedup.pdf}
\caption{Speedup ratio (SheafDB / KG).  LOOKUP: 0.01--0.32$\times$

\subsection{Scalability}

Figure~\ref{fig:scalability} shows latency as a function of dataset size.
Both engines scale linearly for inserts.  SheafDB LOOKUP scales
sub-linearly due to O(1) stalk indexing.

\subsection{Cross-Engine Verification}

All benchmark runs passed cross-engine verification: both KG and
SheafDB produce identical canonical results for every query.
The verification framework compares result sets by fact ID,
subject, relation, objects, and context, confirming semantic
equivalence across representations.

\subsection{Summary of Findings}

\begin{enumerate}
    \item \textbf{Insert:} SheafDB $\approx$ KG.  Both engines scale
          linearly at $\sim$60$\mu$s/fact.
    \item \textbf{LOOKUP:} SheafDB $\gg$ KG.  Stalk-indexed retrieval
          is O(1) vs.\ KG's O(k) triple reconstruction.
    \item \textbf{GLOBAL:} KG $\gg$ SheafDB.  Full scans expose
          SheafDB's open set iteration overhead.
    \item \textbf{Verification:} All cross-engine comparisons pass.
\end{enumerate}

\subsection{Limitations}

The evaluation compares SheafDB against its own KG baseline, not
against production RDF stores (Jena, GraphDB, Virtuoso).  The Jena
adapter has partial rdflib support but is not yet integrated into
the benchmark pipeline.  Results at 10K facts are indicative but
do not demonstrate production-scale behavior.
